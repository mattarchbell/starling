Our approach to finding a solution

Week 9

PHASE 1: Problem Statement and Concept Development

Action: The first step for this project was to define our problem statement, define the ideal scenario if we were able to generate a web solution that solve this problem and investigate the consequences of not having this solution. At the same time we were thinking about what software solutions could be used to tackle and host this web solution.

Outcomes: Initially, the scope of our project was too large. Throughout out development progress we were focused too much on the type of software we would like to use rather then what makes the most sense as a solution to our problem. Below is our initial problem statement submitted to Shawn followed by the problem statement that was approved.

Initial Problem Statement: Create a web map application visualizing data for the conservation authority. Users should be able to filter layers, interact with features using popup displays, and to search for location information. Map will use a combination of data from the conservation authority (e.g. shapefiles obtained directly, information digitized from map images/pdfs) and open-source data (e.g. landcovers, typical monthly precipitation and temperature).

Final problem statement: Landowners can access website to find their property and if it falls into Conservation Authority regulation based on which features exist on their property.

    Development Progress:
  • Our idea was to develop a simple solution for Crowe Valley Conservation Authority that would allow landowners to find wetland information about their property using parcel data and wetland data and determining if those layers overlapped, and assist the conservation authority in rapidly screening development proposals
  • Although a simple idea, we struggled with how to implement this. Originally, we wanted to utilize open source solutions, namely Leaflet, as this would allow the solution to be adopted affordably
  • Geocortex was also explored as a solution. Although a more expensive solution, the program has been used by Ontario counties as a means of effectively disseminating information to residents. Our initial idea was to allow users to export a PDF report for their parcel. We experimented with importing our environmental feature information during a lab session:
The addition of conservation authority vector files to an OpenStreetMap basemap.
The map in the Geocortex HTML viewer.
  • Ultimately, we came to the conclusion that Leaflet would take a lot of effort to learn, not to mention implement, and that Geocortex did not have the functionality that we needed. We decided to use one of the available ESRI web applications, as the CVCA had access to ArcGIS Online
  • Our original problem statement had a massive scope; we intended to allow the user to filter layers, search for data, and we intended to include a ton of layers. Shawn said "no"

Week 10

Project idea development... Round 2

  • Our main idea remained the same, but with a drastically reduced scope. Our new plan of attack was to use ArcGIS Experience Builder to allow land owners to pick a parcel and have a pop-up display to show which layers intersected their land.
  • With approval now garnered from Shawn, we began to explore the functionality of Experience Builder. First we needed data... and a way to display it
  • Our contact at CVCA was gracious enough to send over the parcel, wetlands, waterbodies, watercourses and CA boundary data, which we uploaded to a webmap
  • We struggled more than we probably should have to make sure we all were able to access both the webmap and experience builder application... The group sharing settings are weird, okay?

Week 11

Let's get the party started

  • The first order of business - Lets make this thing look nice! We palyed around with the widgets in experience builder to make it as aesthetic as possible - including the CA logo and a cool side panel to display map info
  • Although I kid, this design process was integral. Making sure the layout was functional was imperative to the user "experience"... We wanted to be sure that anybody accessing this solution would be able to easily navigate through the map

Week 12

Creating the ultimate "Experience"

  • Now with the layout and base functionality of the program done, we had to make it do what we said it would do - allow the user to see if parcels intersected the water features or not...
  • Within the Web Map, we had to configure the pop-ups using Arcade. This language was new to all of us, but we started to figure it out...
  • The benefit to using the Arcade expressions is that it allows the map to query those layers intersecting with the parcel of interest in real time, avoiding time-consuming pre-processing and making the application easy to adapt in the future.
The Arcade expressions used to configure the information displayed on our pop-ups.

Options we looked at

We originally intended to develop this application from scratch... Using open source- leaflet. we considered geocortex... Settled onto creating a web app using arcgis online resources, but which application would we use - Experience builder

What worked

building widgets to costomize the design of our ArcGIS "experience" - worked mostly out of the gate - struggled with clipping of images and title/text, extraneous info from map. Most of the costomization had to be completed in webmaps - arcade expressions to perform the functionality we wanted, making layers non-interactable, etc.

Arcade to the rescue

The main functionality of the program (querying the parcel data) comes from a simple arcade expression that checks to see if the layers we have (wetlands, watercourses, waterbodies, etc. intersect/ are within the boundary of the parcel
